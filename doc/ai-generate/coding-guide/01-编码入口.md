# 编码入口

如果你的目标是**继续写代码而不是重新理解整套方案**，平时优先只读本文件。

## 一、这一层文档解决什么问题

这份入口文档把现有三层设计压缩成日常编码所需的最小上下文：

1. 先知道当前代码骨架已经有什么；
2. 再知道下一步应该优先补哪些真实实现；
3. 最后在需要时跳转到对应的需求、架构、技术栈文档。

## 二、当前后端代码骨架现状

当前 `backend/app/main.py` 已经把这些领域路由接到 FastAPI 主应用：

- `platform`
- `courseware`
- `script`
- `lesson`
- `qa`
- `progress`

当前 `backend/app/` 下已经存在的领域包：

- `common/`：配置、异常、请求上下文、统一响应、签名校验占位
- `platform/`：课程/用户同步
- `courseware/`：解析任务创建与查询
- `parser/`：课件文件信息与结构预览生成
- `cir/`：课程中间表示构建
- `script/`：脚本生成与编辑
- `lesson/`：音频生成、发布、播放装配
- `qa/`：语音转文字、问答、会话查询
- `progress/`：进度跟踪与节奏调整
- `tasks/`：任务记录
- `observability/`：监控、评测、回放能力预留

## 三、当前代码最明显的实现阶段判断

当前代码已经完成了：

- API 路由挂载；
- Pydantic 请求/响应模型拆包；
- 基础成功响应模式；
- 主要业务域的 service 骨架；
- 一部分任务状态记录入口；
- `courseware -> parser -> cir` 的 `.pptx` 结构化 demo 主链；
- 基于 LLM 的 `structurePreview` / `cir` 生成能力（当前仅覆盖解析链）。

当前代码还停留在占位/内存态的部分：

- `verify_signature_placeholder` 仍是占位签名校验；
- `platform/courseware/qa/progress/script/lesson` 主要依赖内存字典或示例返回；
- 尚未真正接入 MySQL / Alembic；
- 尚未真正接入 Dramatiq / Redis 异步执行链；
- 尚未真正接入 MinIO / 对象存储；
- 解析链之外尚未真正接入 LLM / TTS / ASR 外部能力；
- 尚未形成完整测试与联调闭环。

当前解析链需要额外知道：

- `POST /api/v1/lesson/parse` 仍沿用历史接口字段 `fileType="ppt"`，但 demo 实际只接受 `.pptx` 文件
- `parser/pptx_reader.py` 负责抽取 slide 文本，`parser/llm_client.py` 负责调用 OpenAI 兼容接口
- 当前解析是**同步完成**，所以 `taskStatus` 直接返回 `completed`
- 可直接使用 `rest-client/lesson-parse.http` 配合 `examples/demo-courseware.pptx` 验证

## 四、日常编码的默认顺序

不要并行乱补，默认按下面顺序推进：

### 阶段 1：把“解析 demo + 内存态任务”替换成真实解析链路

优先模块：

- `courseware/`
- `parser/`
- `cir/`
- `tasks/`

目标：

- 在现有 `.pptx + LLM` demo 基础上把解析任务从同步内存态升级为真实任务制；
- `ParseTask` 能落 MySQL；
- 课件文件能进入 MinIO；
- 异步解析任务能进入 Dramatiq；
- 查询接口返回真实状态。

### 阶段 2：把脚本、音频、发布链打通

优先模块：

- `script/`
- `lesson/`

目标：

- `ScriptVersion` 进入 MySQL；
- 教师编辑后的版本能保存；
- 音频生成从假数据切成任务制；
- `LessonPackage` 发布态可冻结和查询。

### 阶段 3：把问答从示例回答切成真实证据问答

优先模块：

- `qa/`
- `cir/`
- `parser/`

目标：

- 检索真实页码与证据片段；
- LLM 输出带 `evidencePages` / `evidenceSpans`；
- `QASession` 与 `QARecord` 可持久化；
- 语音转文字具备真实 Provider 接口。

### 阶段 4：把续接与指标链补齐

优先模块：

- `progress/`
- `observability/`

目标：

- `LearningProgress`、`ResumePlan` 可持久化；
- 理解度和续讲决策与 QA 结果联动；
- 样本、耗时、准确率、典型案例可沉淀。

## 六、继续编码前的最小检查清单

- 这次改动属于哪个领域包？
- 是补真实依赖，还是只在示例数据上继续堆逻辑？
- 是否需要同步更新 router / schemas / service / tasks / tests？
- 是否会影响任务状态、发布态、运行态或统一响应结构？
- 是否需要回到上游文档确认对象语义？

如果你已经回答清楚这五个问题，就继续写代码；不要先把整套大文档重新读一遍。
