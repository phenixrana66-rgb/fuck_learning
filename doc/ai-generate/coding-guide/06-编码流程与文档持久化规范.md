# 编码流程与文档持久化规范

本文件用于把“平时怎么开始写、写到一半怎么保持上下文、行为变化后怎么同步文档”固定下来，避免每次都重新建立一套临时规则。

## 一、默认编码流程

每次开始编码，默认按这个顺序：

1. 先读 [01-编码入口.md](./01-编码入口.md)
2. 再读 [03-联动顺序与完成定义.md](./03-联动顺序与完成定义.md)
3. 如果是具体接口任务，再读 [05-接口到代码文件映射.md](./05-接口到代码文件映射.md)
4. 然后进入 [../api/api-navigation.md](../api/api-navigation.md) 和对应接口文档
5. 最后再开始改代码

不要把顺序倒过来。

错误顺序是：

- 直接看接口文档就写代码；
- 只看某个 `service.py` 就继续堆逻辑；
- 写完代码再临时想文档要不要更新。

## 二、为什么要先读 coding-guide，再读 api

- `coding-guide/` 负责告诉你这次改动属于哪个阶段、哪个模块、会影响哪些联动关系；
- `api/` 负责告诉你这个接口的字段和契约细节；
- 两者结合，才能避免“接口字段改了，但模块边界和联动顺序被写乱”。

## 三、文档持久化规则

只要出现下面这些变化，就必须同步更新文档，并把修改真正落盘：

### 1. 接口行为变化

例如：

- 新增字段
- 删除字段
- 字段必填/选填变化
- 响应结构变化
- 错误码语义变化

必须更新：

- `api/` 下对应接口文档

### 2. 代码职责变化

例如：

- 某个模块新增 `repository.py`
- `service.py` 逻辑拆到了 `providers.py` / `tasks.py`
- 某个 domain 的职责边界变了

必须更新：

- `coding-guide/02-后端模块与实现映射.md`
- 如有接口落位变化，再更新 `coding-guide/05-接口到代码文件映射.md`

### 3. 联动顺序变化

例如：

- 原来先接 `qa`，现在必须先打通 `parser/cir`
- 原来同步逻辑变成异步任务
- 发布链路新增新的前置状态

必须更新：

- `coding-guide/03-联动顺序与完成定义.md`

### 4. 当前进度变化

例如：

- 某阶段已完成
- 某个模块已从 demo 切到真实依赖
- 当前阻塞点变化

必须更新：

- `coding-guide/progress/current-status.md`
- 必要时同步更新 `coding-guide/progress/next-start.md`

## 四、最小文档同步原则

不要把每次小改都扩展成大规模文档重写，但也不要让文档失真。

最小原则是：

- **外部可见行为变了，就更新接口文档**
- **内部职责边界变了，就更新 coding-guide 映射文档**
- **阶段完成度变了，就更新 progress 文档**

## 五、每次编码结束前的固定检查

- 我这次是否改了接口行为？
- 我这次是否新增了代码文件或变更了职责落位？
- 我这次是否改变了联动顺序或阶段完成度？
- 我是否已经把这些变化持久化到对应文档？

如果前 3 个问题里任何一个答案是“是”，第 4 个就不能是“否”。

## 六、这套规范的目标

目标不是增加流程负担，而是保证：

- 下次继续开发时不用重新扫全仓；
- 新同学可以先看 coding-guide 再看 api 就快速进入；
- 文档始终和真实代码保持基本同步。
