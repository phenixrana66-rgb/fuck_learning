# 联动顺序与完成定义

本文件回答两个问题：

1. 多个后端模块应该按什么顺序联动；
2. 每个阶段做到什么程度才算真正完成，而不是“接口先返回点假数据”。

## 一、阶段 1：解析链路落真实依赖

涉及模块：

- `common/`
- `courseware/`
- `parser/`
- `cir/`
- `tasks/`

本阶段必须完成：

- 真实签名校验替换 `verify_signature_placeholder`；
- `ParseTask` 从内存字典迁移到 MySQL；
- 原始课件进入 MinIO；
- 解析任务进入 Dramatiq；
- `/lesson/parse` 与 `/lesson/parse/{parseId}` 返回真实任务状态。

当前进度（2026-04）：

- `ParseTask` 主记录这一步已经完成，当前由 `tasks/models.py` + `tasks/repository.py` 落到数据库表 `task_records`；
- `POST /lesson/parse` 已先返回 `processing`，`GET` 可查询 `processing / completed / failed`；
- 但签名校验、MinIO、Dramatiq、课件主记录与解析结果完整落库仍未完成，因此阶段 1 还不能视为完成。

本阶段完成定义（DoD）：

- 重启服务后解析任务不会丢失；
- 同一 `parseId` 可重复查询；
- 失败任务可见失败状态与 `requestId`；
- `structurePreview` 不再完全依赖固定示例。

## 二、阶段 2：脚本、音频、发布态打通

涉及模块：

- `script/`
- `lesson/`
- `tasks/`

本阶段必须完成：

- `ScriptVersion` 可查询、可更新、可版本替换；
- 音频生成改为真实后台任务；
- 发布装配时冻结 `courseware / script / audio` 引用；
- `LessonPackage` 支持 `assembling / published / publish_failed` 之类的可查询状态。

当前进度（2026-04）：

- 脚本主记录已完成首批数据库化，当前通过 `script/models.py` + `script/repository.py` 持久化到 `script_records`；
- `generateScript / getScript / updateScript` 已具备重启后可查询、可更新的最小持久化能力；
- 但音频、发布态仍主要是内存占位，所以阶段 2 仍处于“脚本先行、后半段未完成”的状态。

本阶段完成定义（DoD）：

- 教师修改脚本后不会只停留在内存；
- 发布失败时学生侧读不到伪完整版本；
- 播放接口能基于真实 `LessonPackage` 返回数据。

## 三、阶段 3：问答证据链打通

涉及模块：

- `qa/`
- `cir/`
- `parser/`
- `progress/`

本阶段必须完成：

- `voiceToText` 接入真实 Provider；
- `interact` 不再返回固定示例回答；
- 回答能关联真实页码和证据片段；
- `QASession / QARecord` 可持久化；
- `understandingLevel` 结果能传给后续续讲逻辑。

本阶段完成定义（DoD）：

- 同一 `sessionId` 的多轮历史可查询；
- 证据不足时走保守回答；
- `evidencePages`、`evidenceSpans` 来自真实检索结果。

## 四、阶段 4：续讲与证据输出打通

涉及模块：

- `progress/`
- `observability/`

本阶段必须完成：

- `track` / `adjust` 与真实 QA 结果联动；
- `ResumePlan` 可生成、应用、失效；
- 解析耗时、问答耗时、典型问答案例可输出；
- 样本与指标统计可支持验收和答辩。

本阶段完成定义（DoD）：

- 学生问答后可继续当前章节或回补前置章节；
- 系统可导出最小证据包：耗时统计 + 样本构成 + 典型案例。

## 五、联动时的默认规则

- 先打通状态，再优化内容质量；
- 先保证持久化与可查询，再优化模型效果；
- 先保证教学闭环可跑，再补性能、评测和答辩材料；
- 不要在 placeholder 逻辑上继续堆复杂功能。

## 六、每日编码前后各看一次的检查清单

### 开始前

- 我这次改的是哪个阶段？
- 依赖的上游对象是否已经真实存在？
- 这次改动是否还停留在内存态？

### 结束前

- router / schemas / service / task / 持久化是否一致？
- 状态是否可查询？
- 失败是否可见？
- 是否影响统一响应结构和 `requestId`？

如果这四项没有过，就不要把当前改动视为“阶段完成”。
