# 数据库持久化演进规划

本文件属于 `coding-guide/` 层，目标只有一个：**把当前以内存态为主的 demo 后端，按现有阶段顺序演进为可持久化、可查询、可恢复的真实实现。**

它承接以下上游结论：

- 主数据库已经定为 `MySQL 8.4`；
- 异步与短状态中间层已经定为 `Redis + Dramatiq`；
- 原始课件、解析中间产物、音频等大对象已经定为 `MinIO / S3`；
- 当前仍有多个 `service.py` 使用进程内字典，但 `tasks` 与 `script` 已完成首批真实数据库持久化。

本文件**不重复**技术选型理由，不替代 `api/` 接口契约，也不展开完整物理表设计。它只回答：**下一步数据库和存储层应该先落什么、落到哪里、按什么顺序落。**

## 一、当前存储现状与主要缺口

当前 `backend/app` 仍存在“有的领域已落库、有的领域仍在内存”的混合状态，典型位置包括：

- `backend/app/lesson/service.py`：`_AUDIO_STORE`、`_LESSON_PACKAGES`
- `backend/app/qa/service.py`：`_QA_SESSIONS`
- `backend/app/progress/service.py`：`_PROGRESS_STORE`

当前已经完成首批数据库持久化的位置：

- `backend/app/common/db.py`：统一 engine、session factory、schema 初始化与测试切换入口；
- `backend/app/tasks/models.py` + `backend/app/tasks/repository.py`：解析任务主记录进入 `task_records`；
- `backend/app/script/models.py` + `backend/app/script/repository.py`：脚本主记录进入 `script_records`；
- `backend/app/common/config.py`：已正式消费数据库连接配置，而不是只在文档里声明。

这意味着当前 demo 仍存在几个共同问题：

- `lesson / qa / progress` 仍会在服务重启后丢状态；
- 多实例之间还不能完整共享所有领域状态；
- 任务与发布态还不能全链路回放；
- 失败状态难以追踪；
- 很多接口虽然叫“任务”，但仍未形成完整后台任务制。

因此，数据库规划的首要目标不是“把所有对象一次性建全”，而是先让**状态可查询、失败可见、重启不丢、链路可恢复**。

## 二、当前已完成的首批真实落地（2026-04）

本轮已经完成、并可直接作为后续开发基线的内容有：

- 解析任务主状态已不再依赖进程内 dict，而是统一写入 `task_records`；
- 脚本生成、查询、更新已不再依赖 `_SCRIPT_STORE`，而是统一写入 `script_records`；
- 数据库连接配置已经由代码实际消费，测试可切 SQLite，开发环境可连 MySQL；
- 任务日志仍保留在 `temp/ai-generate/tasks/logs/`，用于补充过程回查，但日志不是数据库主记录的替代品；
- 已完成自动化测试、SQLite 手工验证、MySQL 手工验证，且手工验收使用了真实 `examples/01总论.pptx`。

当前仍未进入本轮范围的部分：

- `lesson` 音频与发布态落库；
- `qa / progress` 持久化；
- MinIO / Redis / Dramatiq / Alembic 的正式接入；
- 真实外部 LLM 全链验证（当前环境缺少 `A12_LLM_API_KEY`，手工验收使用了确定性 patch）。

## 三、存储职责边界

### 1. MySQL：主业务数据库

MySQL 承载需要长期保存、可查询、需要状态流转或版本管理的核心业务对象，包括：

- 课件元数据与解析任务；
- `CIR`、节点树、知识锚点；
- 脚本版本、音频记录、发布快照；
- 问答会话、问答记录；
- 学习进度、续讲计划；
- 与检索相关的部分索引元信息。

数据库层的设计原则是：

- 业务真状态优先落 MySQL；
- 可审计、可重试、可查询的对象优先落 MySQL；
- 不把大文件二进制直接塞进 MySQL。

### 2. Redis：异步与临时状态中间层

Redis 继续承担：

- `Dramatiq` broker；
- 热点结果短缓存；
- 问答短缓存；
- 学习会话临时状态；
- 限流、去重、幂等辅助键。

Redis 不替代 MySQL 保存业务主记录。任何需要在重启后仍可查询的真实任务、脚本、发布、问答、学习进度，都应以 MySQL 为准。

### 3. MinIO / S3：对象文件存储

对象存储继续承载：

- 原始 PPT / PDF 文件；
- 解析中间产物；
- 脚本导出文件；
- 音频文件；
- 其他大对象素材。

数据库中只保存这些对象的元数据、引用关系、状态和定位信息，不直接保存大文件本体。

| 领域对象 | 主存储介质 | 说明 |
|---|---|---|
| `Courseware` / 课件元数据 | MySQL | 保存课件归属、来源、文件引用、解析状态 |
| `ParseTask` | MySQL | 保存解析任务状态、错误信息、结果引用 |
| 原始课件文件 | MinIO / S3 | 保存 `.pptx` / `.pdf` 原文件 |

| 音频文件 | MinIO / S3 | 保存 TTS 产物 |
| 音频任务与音频元数据 | MySQL | 保存生成状态、文件引用、失败原因 |
| `PublishedLessonSnapshot` / `LessonPackage` | MySQL | 保存发布冻结态与播放装配结果 |

## 五、分阶段落库规划

本阶段优先落地：

- 解析失败时能看到失败状态与错误原因；

当前状态补充（2026-04）：

- 上述 4 条里，`ParseTask` 主记录持久化与真实状态查询已经满足；
- 尚未满足的是 `Courseware / ParseResult / KnowledgeNode` 主记录、对象存储引用与真正异步执行链。

### 阶段 2：脚本、音频、发布态落库

涉及模块：`script / lesson / tasks`

本阶段优先落地：

- `ScriptVersion`：草稿、编辑态、确认态、被替换态；
- 音频任务记录与音频文件引用；
- `PublishedLessonSnapshot` / `LessonPackage`：发布冻结态；
- 发布失败状态与装配过程状态。

本阶段完成后至少要满足：

- 教师修改脚本后不会只停留在内存；
- 音频生成与发布状态可查询；
- 学生播放读取真实发布态，而不是示例装配结果。

当前状态补充（2026-04）：

- 第一条已经满足，脚本修改结果可通过数据库重读；
- 第二、三条尚未满足，因为音频与发布态还没有进入真实数据库持久化。

### 阶段 3：问答证据链落库

涉及模块：`qa / cir / parser / progress`

本阶段优先落地：

- `QASession`：会话主记录；
- `QARecord`：问题、回答、证据页码、证据片段、理解度；
- 与 `CIR`、页码证据的引用关系；
- 适合放 Redis 的短缓存与会话临时态。

本阶段完成后至少要满足：

- 同一 `sessionId` 的多轮历史可查询；
- 证据链可回放；
- 理解度结果可传给后续续讲逻辑。

### 阶段 4：进度与续讲持久化

涉及模块：`progress / observability`

本阶段优先落地：

- `LearningProgress`：节点进度、掌握度、更新时间；
- `ResumePlan`：续讲建议、应用状态、失效时间；
- 关键观测结果的统计沉淀。

本阶段完成后至少要满足：

- 学习进度可跨会话恢复；
- 续讲建议可查询、可应用、可失效；
- 验收时可导出最小证据包。

## 七、接口与状态语义需要同步演进

数据库落地不只是把 dict 改成表，还会改变接口状态语义。当前最需要注意的是：

- `POST /api/v1/lesson/parse` 当前已经改为先返回 `processing`，再由查询接口反映真实完成态；
- 当前接口仍要继续保持 `pending / processing / completed / failed` 等真实状态语义，并为正式异步执行链预留一致口径；
- `GET /api/v1/lesson/parse/{parseId}` 要返回真实进度与失败原因，而不是从内存对象直接拼装；
- 当前接口字段里仍有 `fileType="ppt"`，但 demo 实际只接受 `.pptx`，后续落真实解析策略时要同步收口语义。

换句话说，数据库演进会反向影响：

- `router.py` 的返回结构；
- `schemas.py` 的状态字段；
- `service.py` 的同步/异步编排方式；
- `api/` 下的接口契约文档。

## 八、代码落位与工程规则

后续真正开始接真实持久化时，默认遵循以下落位：

- 领域对象模型：放对应领域的 `models.py`
- 数据访问：放对应领域的 `repository.py`
- 迁移脚本：统一放 `backend/alembic/`
- 任务触发：仍由领域 `service.py` 发起
- 后台执行：放 `backend/app/tasks/` 或领域下的 `tasks.py`

默认规则：

- router 不直接操作数据库；
- service 负责领域编排，不继续堆模块级 dict；
- repository 负责数据库读写边界；
- 大对象文件只保存引用，不直接落数据库本体。

## 十一、当前数据库基线补充（2026-04）

当前仓库已不再停留在“本地文件桥接层即主状态源”的阶段，而是形成了一个**数据库主记录 + temp 日志补充**的最小基线：

- `backend/app/tasks/repository.py`：当前负责把 `TaskRecord` 持久化到数据库表 `task_records`；
- `backend/app/script/repository.py`：当前负责把脚本主记录持久化到数据库表 `script_records`；
- `temp/ai-generate/tasks/logs/{taskId}.log`：继续保存任务过程日志；
- 测试环境通常使用 SQLite 文件库，开发/手工验收环境可切到 MySQL。

这个基线的定位是：

- 先让 parse/task 与 script 具备最小真实持久化能力；
- 先让重启后查询、失败态回看、脚本更新回读成立；
- 为后续补齐 `courseware / lesson / qa / progress` 的 models / repository / migration 留出稳定落位。

它仍然**不是**完整完成态，也不替代：

- Alembic 迁移脚本；
- Redis / Dramatiq 的正式异步执行链；
- MinIO / S3 原始文件与大对象存储；
- lesson / qa / progress 的真实数据库主记录。

## 十二、当前可直接用于开发的最小表清单（2026-04 补充）

### 1. 阶段 1：解析链路首批落表

建议先固定以下 5 张表：

| 表名 | 当前开发用途 | 对应代码域 |
|---|---|---|

| `chapter_ppt_assets` | 原始 PPT 元数据、版本号、页数、上传状态 | `courseware/` |

| `student_section_progress` | 章节进度、掌握度、最近问答 | `progress/` |

当前对接时最值得直接采用的字段包括：

- `qa_answers.understanding_level`：与现有 `progress/adjust` 语义直接对接
- `qa_answers.recommended_section_id`、`recommended_anchor_id`：为续讲落点提供直接引用
- `student_lesson_progress.current_unit_id`、`current_section_id`、`current_anchor_id`：保证跨会话恢复时能回到正确位置
- `student_section_progress.last_qa_message_id`、`mastery_percent`：为问答影响学习节奏提供最小证据面
