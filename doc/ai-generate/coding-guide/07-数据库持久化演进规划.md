# 数据库持久化演进规划

本文件属于 `coding-guide/` 层，目标只有一个：**把当前以内存态为主的 demo 后端，按现有阶段顺序演进为可持久化、可查询、可恢复的真实实现。**

它承接以下上游结论：

- 主数据库已经定为 `MySQL 8.4`；
- 异步与短状态中间层已经定为 `Redis + Dramatiq`；
- 原始课件、解析中间产物、音频等大对象已经定为 `MinIO / S3`；
- 当前多个 `service.py` 仍在使用进程内字典，尚未具备真实持久化能力。

本文件**不重复**技术选型理由，不替代 `api/` 接口契约，也不展开完整物理表设计。它只回答：**下一步数据库和存储层应该先落什么、落到哪里、按什么顺序落。**

## 一、当前存储现状与主要缺口

当前 `backend/app` 的多个领域仍处于“接口已通、状态仍在内存”的阶段，典型位置包括：

- `backend/app/courseware/service.py`：`_PARSE_TASKS`
- `backend/app/tasks/service.py`：`_TASKS`
- `backend/app/script/service.py`：`_SCRIPT_STORE`
- `backend/app/lesson/service.py`：`_AUDIO_STORE`、`_LESSON_PACKAGES`
- `backend/app/qa/service.py`：`_QA_SESSIONS`
- `backend/app/progress/service.py`：`_PROGRESS_STORE`

这意味着当前 demo 存在几个共同问题：

- 服务重启后状态丢失；
- 多实例之间无法共享状态；
- 任务与发布态无法回放；
- 失败状态难以追踪；
- 很多接口虽然叫“任务”，但本质仍是同步返回或示例返回。

因此，数据库规划的首要目标不是“把所有对象一次性建全”，而是先让**状态可查询、失败可见、重启不丢、链路可恢复**。

## 二、存储职责边界

### 1. MySQL：主业务数据库

MySQL 承载需要长期保存、可查询、需要状态流转或版本管理的核心业务对象，包括：

- 课件元数据与解析任务；
- `CIR`、节点树、知识锚点；
- 脚本版本、音频记录、发布快照；
- 问答会话、问答记录；
- 学习进度、续讲计划；
- 与检索相关的部分索引元信息。

数据库层的设计原则是：

- 业务真状态优先落 MySQL；
- 可审计、可重试、可查询的对象优先落 MySQL；
- 不把大文件二进制直接塞进 MySQL。

### 2. Redis：异步与临时状态中间层

Redis 继续承担：

- `Dramatiq` broker；
- 热点结果短缓存；
- 问答短缓存；
- 学习会话临时状态；
- 限流、去重、幂等辅助键。

Redis 不替代 MySQL 保存业务主记录。任何需要在重启后仍可查询的真实任务、脚本、发布、问答、学习进度，都应以 MySQL 为准。

### 3. MinIO / S3：对象文件存储

对象存储继续承载：

- 原始 PPT / PDF 文件；
- 解析中间产物；
- 脚本导出文件；
- 音频文件；
- 其他大对象素材。

数据库中只保存这些对象的元数据、引用关系、状态和定位信息，不直接保存大文件本体。

## 三、核心对象与存储介质映射

| 领域对象 | 主存储介质 | 说明 |
|---|---|---|
| `Courseware` / 课件元数据 | MySQL | 保存课件归属、来源、文件引用、解析状态 |
| `ParseTask` | MySQL | 保存解析任务状态、错误信息、结果引用 |
| 原始课件文件 | MinIO / S3 | 保存 `.pptx` / `.pdf` 原文件 |
| `StructurePreview` / `CIR` / 节点树 / 锚点 | MySQL | 保存结构化结果与教学中间表示 |
| `ScriptVersion` | MySQL | 保存脚本版本、编辑态、确认态 |
| 音频文件 | MinIO / S3 | 保存 TTS 产物 |
| 音频任务与音频元数据 | MySQL | 保存生成状态、文件引用、失败原因 |
| `PublishedLessonSnapshot` / `LessonPackage` | MySQL | 保存发布冻结态与播放装配结果 |
| `QASession` / `QARecord` | MySQL | 保存会话、问题、回答、证据、理解度 |
| 问答短缓存 / 临时会话态 | Redis | 保存短期缓存与运行时临时态 |
| `LearningProgress` / `ResumePlan` | MySQL | 保存学习进度与续讲决策 |

## 四、分阶段落库规划

本节默认对齐 [03-联动顺序与完成定义.md](./03-联动顺序与完成定义.md)。

### 阶段 1：解析链路先真实化

涉及模块：`courseware / parser / cir / tasks`

本阶段优先落地：

- `Courseware`：课件元数据、文件引用、解析来源；
- `ParseTask`：任务状态、错误信息、开始/结束时间、结果引用；
- `TaskRecord`：统一任务元信息；
- `StructurePreview` / `CIR`：解析结果；
- 原始课件文件对象引用：落 MinIO / S3；
- 解析任务执行：从同步 demo 改为真实任务制。

本阶段完成后至少要满足：

- 服务重启后 `parseId` 仍可查询；
- 解析失败时能看到失败状态与错误原因；
- `POST /api/v1/lesson/parse` 不再默认同步完成后直接 `completed`；
- `GET /api/v1/lesson/parse/{parseId}` 返回真实任务状态。

### 阶段 2：脚本、音频、发布态落库

涉及模块：`script / lesson / tasks`

本阶段优先落地：

- `ScriptVersion`：草稿、编辑态、确认态、被替换态；
- 音频任务记录与音频文件引用；
- `PublishedLessonSnapshot` / `LessonPackage`：发布冻结态；
- 发布失败状态与装配过程状态。

本阶段完成后至少要满足：

- 教师修改脚本后不会只停留在内存；
- 音频生成与发布状态可查询；
- 学生播放读取真实发布态，而不是示例装配结果。

### 阶段 3：问答证据链落库

涉及模块：`qa / cir / parser / progress`

本阶段优先落地：

- `QASession`：会话主记录；
- `QARecord`：问题、回答、证据页码、证据片段、理解度；
- 与 `CIR`、页码证据的引用关系；
- 适合放 Redis 的短缓存与会话临时态。

本阶段完成后至少要满足：

- 同一 `sessionId` 的多轮历史可查询；
- 证据链可回放；
- 理解度结果可传给后续续讲逻辑。

### 阶段 4：进度与续讲持久化

涉及模块：`progress / observability`

本阶段优先落地：

- `LearningProgress`：节点进度、掌握度、更新时间；
- `ResumePlan`：续讲建议、应用状态、失效时间；
- 关键观测结果的统计沉淀。

本阶段完成后至少要满足：

- 学习进度可跨会话恢复；
- 续讲建议可查询、可应用、可失效；
- 验收时可导出最小证据包。

## 六、接口与状态语义需要同步演进

数据库落地不只是把 dict 改成表，还会改变接口状态语义。当前最需要注意的是：

- `POST /api/v1/lesson/parse` 当前在 demo 中同步完成，因此 `taskStatus` 直接是 `completed`；
- 落库并改为任务制后，接口要支持真实的 `pending / processing / completed / failed` 等状态；
- `GET /api/v1/lesson/parse/{parseId}` 要返回真实进度与失败原因，而不是从内存对象直接拼装；
- 当前接口字段里仍有 `fileType="ppt"`，但 demo 实际只接受 `.pptx`，后续落真实解析策略时要同步收口语义。

换句话说，数据库演进会反向影响：

- `router.py` 的返回结构；
- `schemas.py` 的状态字段；
- `service.py` 的同步/异步编排方式；
- `api/` 下的接口契约文档。

## 七、代码落位与工程规则

后续真正开始接真实持久化时，默认遵循以下落位：

- 领域对象模型：放对应领域的 `models.py`
- 数据访问：放对应领域的 `repository.py`
- 迁移脚本：统一放 `backend/alembic/`
- 任务触发：仍由领域 `service.py` 发起
- 后台执行：放 `backend/app/tasks/` 或领域下的 `tasks.py`

默认规则：

- router 不直接操作数据库；
- service 负责领域编排，不继续堆模块级 dict；
- repository 负责数据库读写边界；
- 大对象文件只保存引用，不直接落数据库本体。

## 十、当前桥接状态补充（2026-04）

在正式接入 MySQL / Redis / Dramatiq 前，当前仓库已增加一个**本地文件桥接层**，用于减少解析任务完全依赖进程内字典的问题：

- `backend/app/tasks/repository.py`：负责把 `TaskRecord` 写入本地文件；
- 当前默认文件位置：`temp/ai-generate/tasks/tasks.json`；
- 当前默认日志位置：`temp/ai-generate/tasks/logs/{taskId}.log`。

这个桥接层的定位是：

- 让当前解析任务在服务重启后仍可恢复查询；
- 让失败态、`requestId`、完成时间等元信息先具备最小持久化；
- 为后续真正迁移到 `tasks/repository.py + MySQL` 提前稳定代码落位。

它**不是**阶段 1 的最终完成态，也不替代：

- MySQL 主记录；
- Redis / Dramatiq 异步执行；
- MinIO / S3 原文件存储。
