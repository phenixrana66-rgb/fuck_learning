# 后端模块与实现映射

本文件把当前 `backend/app` 现有代码骨架与后续实现目标做一一映射，减少开发时在架构文档和代码目录之间来回翻译。

## 一、总映射原则

当前后端普遍采用同一模式：

- `router.py`：HTTP 接口入口
- `schemas.py`：请求/响应 DTO
- `service.py`：业务编排与领域逻辑

后续需要逐步补上的附加层：

- `repository.py`：数据库访问（如开始接 SQLAlchemy）
- `models.py`：ORM 模型
- `tasks/*.py`：异步任务
- `providers/*.py`：外部能力调用（LLM / TTS / ASR / 对象存储）
- `tests/`：模块测试与接口测试

## 二、按模块映射

### 1. `common/`

当前职责：

- 配置
- 统一异常
- 请求上下文
- 统一成功响应
- 签名校验占位

后续重点：

- 把签名校验从 placeholder 切为真实算法；
- 统一 `requestId`、错误码、异常映射；
- 把安全、时间、日志上下文收口到公共层。

### 2. `platform/`

当前现状：

- `router.py` 已暴露 `syncCourse` / `syncUser`；
- `service.py` 仍是示例映射与 token 占位。

后续重点：

- 建立外部 ID 与内部 ID 映射表；
- 接入真实签名校验与鉴权；
- 明确幂等策略和重复同步行为。

### 3. `courseware/` + `parser/` + `cir/`

当前现状：

- `courseware/service.py` 能创建解析任务并返回预览；
- 解析任务状态已统一收口到 `tasks/service.py` 的 `_TASKS`，不再单独维护 `_PARSE_TASKS`；
- `parser/pptx_reader.py` 已能读取本地或远程 `.pptx`，提取 slide 标题、正文、表格和备注；
- `parser/llm_client.py` 已能调用 OpenAI 兼容接口生成章节结构；
- `parser/service.py` 已能把 `.pptx` 提取结果转换成 `StructurePreview`；
- `cir/service.py` 已能基于 preview 生成页码驱动的 CIR；
- 当前 `POST /lesson/parse` 已改为先提交任务并返回 `processing`，解析执行在本地后台路径继续完成；
- 当前解析失败不再由 `POST` 同步返回，而是通过查询接口暴露失败态；
- 解析任务记录已可通过 `tasks/repository.py` 落到本地文件，作为接入真实 MySQL 前的桥接持久化；
- 当前 demo 仍只支持 `.pptx`，不支持 `.pdf` 与旧 `.ppt`。

后续重点：

- 解析任务落库；
- 课件文件落对象存储；
- 增加 `.pdf` 与旧 `.ppt` 的解析策略；
- 生成更细粒度的 CIR、KnowledgeAnchor、页码证据；
- 解析、索引构建、检索辅助数据构建等任务改为异步执行。

### 4. `script/`

当前现状：

- 已有脚本生成、查询、更新骨架；
- `_SCRIPT_STORE` 仍是内存态；
- 版本控制只有基础字段递增；
- `generateScript` 已开始读取真实 `parseId` 对应的解析结果，并基于 `CIR` 生成讲稿段落。

后续重点：

- 引入 `ScriptVersion` 数据模型；
- 区分 `draft / teacher_editing / confirmed / superseded`；
- 让脚本生成与 CIR 真正联动。

### 5. `lesson/`

当前现状：

- 已有生成音频、发布、播放装配接口；
- `_AUDIO_STORE`、`_LESSON_PACKAGES` 仍是内存态；
- 已接入任务记录入口，但还不是实际异步执行；
- `generateAudio -> publish -> play` 已开始消费真实脚本、课件与快照引用，能够组成最小完整 demo 链。

后续重点：

- 音频生成改为任务制；
- `LessonPackage` 发布快照落库；
- 发布失败、资源缺失、装配中状态可查询；
- 播放接口基于真实发布态输出。

### 6. `qa/`

当前现状：

- 已有 `voiceToText`、`interact`、`session` 接口；
- `_QA_SESSIONS` 仍是内存态；
- 当前回答为受控示例文本，不是实际检索+生成链。

后续重点：

- 接入真实 ASR Provider；
- 建立检索层与证据片段回链；
- 让 `answerConfidence`、`evidencePages`、`evidenceSpans` 来自真实逻辑；
- 会话与记录持久化。

### 7. `progress/`

当前现状：

- 已有 `track` / `adjust` 接口；
- `_PROGRESS_STORE` 仍是内存态；
- 续讲规则仍是基础 if/else 决策。

后续重点：

- `LearningProgress`、`ResumePlan` 持久化；
- 与 `QARecord` 真实联动；
- 把理解度与前置节点判断从示例规则升级为可追踪策略。

### 8. `tasks/`

当前职责：

- 统一记录异步任务元信息，并已开始承载解析任务的状态、结果、失败信息与本地桥接持久化。

后续重点：

- 接入 Dramatiq actor；
- 区分解析、音频、发布装配、索引构建等任务；
- 继续让 API 查询逐步切到任务真实状态，而不是模拟状态。

### 9. `observability/`

当前职责：

- 为耗时、样本、案例、评测与回放预留位置。

后续重点：

- 沉淀解析耗时、问答耗时、错误率；
- 输出样本统计、典型问答案例、证据回链结果；
- 服务答辩证据与回归验证。

## 三、代码实现时的默认落位方式

### HTTP 层

- 新增接口：优先放到对应领域包的 `router.py`
- 入参/出参：放到该领域的 `schemas.py`

### 业务层

- 领域编排：先放 `service.py`
- 当 `service.py` 过重时，再拆 `repository.py`、`providers.py`、`policies.py`

### 数据层

- 持久化对象：优先按领域放 `models.py`
- 迁移：放到 `backend/alembic/`

### 异步层

- 真正的后台任务：放 `backend/app/tasks/` 或领域包下的 `tasks.py`
- 任务触发仍由领域 `service.py` 发起，不让 router 直接编排队列细节

## 四、这份文档不做什么

- 不重复需求背景；
- 不重复总体架构设计；
- 不替代详细 API 文档；
- 不强行定义所有实现细节。

它只做一件事：**让你看到代码目录时，知道下一步每个模块应该补什么。**
