# 业务流程与学习链路

## 总体链路概览

本项目的业务主线可以概括为：

课件接入 -> 结构化解析 -> 智课生成 -> 教师确认与发布 -> 学生学习 -> 实时问答 -> 进度跟踪 -> 续讲或补讲

这条链路必须是闭环的，不能只做前半段的内容生成，也不能只做后半段的问答交互。

## 一、教师侧业务流程

### 1. 上传课件

教师选择课程并上传 PPT 或 PDF 课件，系统记录文件信息、课程关联和发起人信息。

### 2. 发起解析任务

系统创建异步解析任务，对课件做页级抽取、知识点预览、章节或子章节结构整理。

### 3. 查看结构预览

教师查看结构化结果，重点关注：

- 章节层级是否合理。
- 重点识别是否准确。
- 页码与知识点是否有明显错位。

### 4. 生成讲稿

系统基于解析结果生成结构化讲稿，讲稿应包含开场、讲解和过渡内容，并尽量保持教学逻辑连续。

### 5. 教师编辑与确认

教师对讲稿进行调整，修正不合适表达，确认后进入音频生成和发布流程。

### 6. 生成音频与发布智课

系统根据脚本生成章节音频，并将课程结构、脚本和音频打包为正式智课版本，供学生端学习。

## 二、学生侧学习流程

### 1. 进入智课

学生通过平台入口进入当前课程的智课内容，系统获取课程、用户和当前学习上下文。

### 2. 按章节学习

学生按章节或节点开始学习，系统展示当前章节、进度和讲授内容，并播放对应音频或文本内容。

### 3. 发起问题

学生在学习过程中可以随时提问：

- 文字提问直接进入问答链路。
- 语音提问先转文字，再进入问答链路。

### 4. 获取回答

系统结合当前学习章节、历史问答和课件结构，输出回答内容，并尽量返回关联知识点与页码证据。

### 5. 判断理解程度

系统根据问题和回答情况给出理解程度判断，例如未理解、部分理解、完全理解。

### 6. 调整后续学习

基于理解程度，系统选择：

- 正常继续当前链路。
- 插入补讲内容。
- 回到前置节点后再继续。

## 三、平台交互流程

平台在整个链路中不仅是承载环境，也承担对象同步和调用入口角色：

- 通过课程同步接口提供课程映射。
- 通过用户同步接口提供教师或学生身份信息。
- 通过开放 API 调用解析、问答、进度等能力。

## 四、关键流程断点与需求含义

### 断点 1，解析后没有结构

如果系统只能抽文本，后续讲稿、问答和续讲都会失去统一上下文，因此必须有课程结构化结果。

### 断点 2，问答脱离当前内容

如果问答不受当前章节约束，就会退化为普通聊天机器人，无法体现教学相关性。

### 断点 3，问后无法继续讲

如果回答完就停住，学习链路仍然是断的，所以进度跟踪和续接策略是主链路必备，而不是附加增强。

## 五、需求分析中的流程结论

整个系统的最小可行闭环不是“上传课件后得到一段文案”，而是：

上传课件后生成可编辑智课，学生学习时可提问，系统回答后还能继续讲，并对学习进度和理解情况产生可追踪结果。
