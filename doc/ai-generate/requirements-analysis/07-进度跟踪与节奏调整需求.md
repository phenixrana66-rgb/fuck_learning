# 进度跟踪与节奏调整需求

## 模块定位

进度跟踪与节奏调整模块负责解决“问完之后怎么继续讲”的问题。它使系统从“会回答”提升到“能维持教学连续性”。

## 一、进度跟踪需求

### 1. 跟踪对象

系统应至少跟踪：

- 当前智课 ID
- 当前学习章节 ID
- 当前章节学习进度百分比
- 最近操作时间
- 最近问答记录 ID

### 2. 跟踪结果

进度接口至少应返回：

- trackId
- totalProgress
- nextSectionSuggest

### 3. 业务意义

进度跟踪不是简单埋点，而是后续节奏判断、章节建议和答辩统计的输入基础。

## 二、理解程度判断需求

### 1. 理解程度来源

理解程度应结合问答结果产生，至少支持：

- none
- partial
- full

### 2. 使用目的

理解程度用于驱动后续节奏调整，而不是仅做展示字段。

## 三、节奏调整需求

### 1. 输入条件

节奏调整至少应关联：

- userId
- lessonId
- currentSectionId
- understandingLevel
- qaRecordId

### 2. 输出内容

节奏调整结果至少应包括：

- continueSectionId
- adjustType
- supplementContent
- nextSections

### 3. 调整类型

根据来源材料，当前轮至少要覆盖以下三类决策意图：

- 正常继续
- 补充讲解
- 依据前置关系回补后继续

开放 API 文档给出的 `normal`、`supplement`、`accelerate` 可作为接口表现形式，需求层则重点强调教学决策含义。

## 四、补讲需求

当学生对当前知识点只部分理解，系统应支持插入补讲内容，补讲至少应满足：

- 内容仍围绕当前知识点。
- 时长相对简短，可插入原链路中。
- 尽量配合例子、解释或强化说明。

## 五、回到原节点继续讲的需求

该项是赛题差异化重点之一。系统应保证：

- 学生提问后，不会失去当前讲授位置。
- 回答或补讲结束后，可以回到原章节继续讲。
- 如果需要回到前置节点，也应明确后续再回到当前节点。

## 六、可视化与感知需求

虽然需求层不展开具体页面设计，但当前轮至少要求：

- 学习进度具备可视化表达。
- 节奏调整结果对学生侧和演示侧可感知。
- 能支持展示“当前在哪一节、因为什么补讲、接下来去哪一节”。

## 七、性能与稳定性要求

- 节奏调整应嵌入问答后链路，不能形成明显等待断层。
- 进度记录应尽量轻量，不影响主学习体验。

## 八、需求结论

如果没有进度跟踪与节奏调整模块，本项目只能算“会生成内容和回答问题”，不能算“能完成教学闭环”。因此该模块属于当前轮主线必做能力。
