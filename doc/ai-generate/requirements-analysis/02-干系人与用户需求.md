# 干系人与用户需求

## 干系人总览

本项目至少涉及三类核心干系人：

- 教师
- 学生
- 平台方

除此之外，还存在评审方、项目团队和后续运维支持方，但本目录以直接影响需求边界的三类角色为主。

## 教师侧需求

教师是智课生成链路的发起者，也是内容质量与教学表达的最终把关者。教师侧的核心需求如下。

### 1. 课件输入要稳定

- 支持 PPT、PDF 课件接入。
- 能识别知识点、章节层级、重点内容、公式图表等关键元素。
- 解析结果应尽量结构化，而不是只给出大段文本。

### 2. 讲稿生成要可用

- 能生成包含开场白、讲解、过渡语的结构化讲稿。
- 讲稿要尽量符合教学逻辑，不是把原文简单拼接。
- 教师需要保留编辑权，能调整措辞、删改不合适内容。

### 3. 内容发布要可控

- 支持语音合成并形成可播放资源。
- 支持确认后发布，避免学生端直接读取未审核版本。
- 能看到结构预览、脚本版本和可能的重点遗漏。

### 4. 效果证据要可展示

- 能沉淀解析样例、讲稿样例、结构树、指标统计等材料。
- 便于后续提交方案、PPT、演示视频和答辩说明。

## 学生侧需求

学生是学习闭环的直接使用者。学生关注的不是后台结构，而是“学得顺不顺、问得准不准、能不能继续讲”。

### 1. 学习入口要轻量

- 适配平台 Web 端或移动端嵌入。
- 页面应以 H5 / WebView 友好方式运行。
- 在常规设备和普通网络条件下保持可用。

### 2. 问答要与当前内容强相关

- 支持文字提问。
- 支持语音提问，并能先转为文字后进入问答。
- 支持多轮互动，但回答不能脱离当前学习上下文乱扩展。

### 3. 回答要可信

- 尽量说明答案关联到哪个知识点、章节或页码。
- 对课件无依据的问题应保守处理，而不是编造答案。
- 回答速度要足够快，不能明显打断学习节奏。

### 4. 学习过程要连续

- 提问后能回到刚才讲到的位置继续学习。
- 没听懂时能获得补讲或更慢的节奏。
- 学习进度能被记录，并可据此给出后续建议。

## 平台方需求

平台方关注的是接入成本、接口规范、安全边界和后续扩展性。

### 1. 接口要标准化

- 具备版本号管理，例如 `/api/v1`。
- 请求与响应采用统一 JSON 结构。
- 至少提供 10 个功能接口，覆盖关键业务链路。

### 2. 平台对象要可同步

- 支持课程信息同步。
- 支持用户信息同步。
- 支持课程、用户与智课对象的关联映射。

### 3. 安全要可管控

- 支持 `enc` 签名校验。
- 支持 HTTPS、权限控制、日志审计。
- 支持请求排障所需的 `requestId`。

### 4. 能力要可扩展

- 接口新增字段时不应轻易破坏兼容性。
- 后续可以扩展更多多模态能力。
- `数字人` 可以预留为扩展方向，但当前轮不作为强制交付。

## 需求冲突与平衡原则

三类角色之间天然存在拉扯，需求分析必须提前定下平衡原则：

- 教师要可控，学生要顺滑，平台要标准，三者不能只满足其一。
- 当前轮优先满足教学闭环与平台标准化，不为了增强表现牺牲主链路稳定性。
- 所有角色需求中，以赛题明示指标和主链路要求为最高优先级。
