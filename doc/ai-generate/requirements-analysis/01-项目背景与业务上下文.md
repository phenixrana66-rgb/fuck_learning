# 项目背景与业务上下文

## 项目背景

本项目面向高校通识课自主学习场景，核心目标不是把 PPT 或 PDF 转成可以播放的静态内容，而是构建一条完整的 AI 互动教学闭环。赛题材料明确指出，当前教学智能化落地中存在三类连续性问题：

- 课件解析多数停留在文本抽取层，难以识别知识点逻辑、公式图表和讲授顺序。
- AI 讲授缺少结构化脚本，内容容易零散、跳跃，缺少开场、过渡和重点组织。
- 学生提问后，系统常常无法结合当前课件上下文精准回答，也难以回到原来的讲授节点继续讲。

因此，本项目要解决的不是单点能力问题，而是“课件解析、智课生成、实时问答、进度续接”之间的断裂问题。

## 业务目标

从业务角度看，本项目需要形成以下结果：

1. 教师上传课件后，系统能够识别课程结构与重点，生成可编辑的讲授脚本。
2. 学生在学习过程中可以通过文字或语音提问，系统结合当前智课上下文进行回答。
3. 问答结束后，系统能够识别学生理解程度，选择继续讲、补讲或回到前置节点重讲。
4. 整个过程可以嵌入泛雅 Web 端或学习通移动端，满足平台集成和开放接口要求。

## 业务价值

### 对教师的价值

- 降低从课件到讲授脚本的准备成本。
- 保留教师对讲稿的编辑权，避免完全失控的机器生成。
- 提供结构化讲解资源和可复用音频资产，便于后续复用与迭代。

### 对学生的价值

- 把静态课件转成更连续、更可打断、可继续的学习体验。
- 在当前学习章节下获得更高相关度的回答，而不是泛化闲聊。
- 在没听懂时获得补讲或节奏调整，而不是被动跳到下一页。

### 对平台方的价值

- 形成可集成、可扩展的教学智能能力，而不是孤立 Demo。
- 通过开放 API 方式承接课程同步、用户同步、问答、进度等业务链路。
- 为后续课程运营、数据统计和平台扩展保留标准入口。

## 项目所处的业务环境

本项目运行在教育平台内嵌场景下，意味着它不是独立消费类应用，而是一个面向教学流程的嵌入式能力模块。由此带来三个上下文约束：

- 系统需要兼容平台身份、课程体系、终端环境和接口调用方式。
- 系统不仅要能生成内容，还要能被平台调用、追踪和审计。
- 系统的好坏最终由教学适配性、落地可行性、性能和可交付证据共同决定。

## 项目成功的业务定义

如果用一句话概括成功标准，本项目应被定义为：

一个能够在泛雅平台场景下，把课件转成可讲、可问、可继续讲的互动智课系统，并且其核心链路有明确指标、有标准接口、有可验收的交付证据。
