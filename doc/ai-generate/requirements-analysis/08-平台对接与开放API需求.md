# 平台对接与开放API需求

## 模块定位

平台对接与开放 API 需求决定了系统不能只做内部 Demo，而必须形成一套可供教育平台调用、嵌入和扩展的标准化能力接口。

## 一、接口体系需求

### 1. 版本化要求

- 接口 URL 应包含版本号。
- 当前规范以 `/api/v1` 为基线。

### 2. 协议与格式要求

- 采用 HTTP / HTTPS，推荐 HTTPS。
- 请求与响应统一采用 UTF-8 JSON。

### 3. 响应结构要求

所有对外接口应遵循统一结构：

- code
- msg
- data
- requestId

## 二、核心接口覆盖需求

当前轮应至少覆盖以下能力分组：

- 课件上传与解析
- 智课脚本生成
- 语音合成
- 问答交互
- 语音转文字
- 学习进度追踪
- 学习节奏调整
- 课程同步
- 用户同步

结合赛题要求，功能接口总数不得少于 10 个。

## 三、异步调用需求

赛题明确要求课件解析支持异步调用集成，因此应至少满足：

- 提交解析请求后返回任务标识。
- 可以查询任务状态。
- 处理结果可供平台后续拉取。

从需求角度看，以下能力也应优先按异步模式设计：

- 课件解析
- 音频生成
- 正式发布装配

## 四、平台同步需求

### 1. 课程同步

- 平台方应能同步课程基础信息。
- 系统应建立外部课程 ID 与内部课程对象的映射。

### 2. 用户同步

- 平台方应能同步教师与学生用户信息。
- 系统应识别角色并完成内部身份映射。

### 3. 权限与会话承接

- 接口需支持平台侧身份校验需求。
- 需保留平台内嵌场景下的身份承接能力。

## 五、兼容性需求

开放 API 文档明确要求兼容演进，因此需求层应遵循：

- 新增字段尽量采用非必填扩展字段。
- 次版本新增功能时不轻易破坏原有参数与返回结构。
- 重大升级需保留迁移空间。

## 六、接口文档与交付需求

- 接口应有可说明、可展示的文档化结果。
- 接口能力应能支撑答辩中“如何与泛雅平台集成”的说明。
- 接口规划本身属于正式交付和验收证据的一部分。

## 七、与需求主线的关系

平台与 API 需求不是外围附录，而是当前轮主线约束之一。原因有三：

1. 赛题明确要求平台嵌入和异步集成。
2. 开放 API 规范给出了明确的接口基线。
3. 没有标准接口，本项目无法被说明为真正可落地的教育平台能力。

## 八、需求结论

本项目必须形成覆盖教学闭环的开放 API 能力体系，并与课程同步、用户同步、安全签名、版本兼容要求共同组成平台对接基线。
