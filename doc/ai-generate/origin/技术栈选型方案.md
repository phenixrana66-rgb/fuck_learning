# AI互动智课项目技术栈选型方案

## 1. 选型结论

本项目建议采用 **Python 作为核心后端语言**，整体走 **模块化单体 + 异步任务 + 对象存储 + 混合检索 + 托管式 LLM/TTS 服务** 的技术路线。  
对 4 人团队、5 周周期、Demo 优先的目标而言，这条路线复杂度最低，但能够覆盖当前正式架构方案中的关键能力：

- 课件解析与 CIR 结构化
- LLM 编排与受控问答
- TTS 音频生成与播放装配
- 发布快照与学习运行态
- 进度续接与评测观测

本项目**不建议**一开始做微服务、大规模 MLOps、自建模型服务或复杂 Agent 框架。

---

## 2. 总体技术路线

```text
前端：React SPA
    ↓
后端：FastAPI 主服务
    ↓
异步：Dramatiq Worker
    ↓
存储：PostgreSQL + Redis + S3/MinIO
    ↓
智能：托管 LLM API + 托管 TTS API
    ↓
检索：Postgres Full-Text + pgvector + 重排
```

系统运行形态建议固定为：

- **一个代码仓库**
- **两个核心进程**：`web` + `worker`
- **一套主数据库**：PostgreSQL
- **一个缓存/任务中间件**：Redis
- **一个对象存储**：S3 兼容存储

---

## 3. 后端技术栈

### 3.1 核心语言

- **Python 3.11+**

选择原因：

- 适合 PPT/PDF 解析生态
- 适合 LLM SDK、Prompt 编排、评测脚本
- 适合快速搭建 API 与异步任务
- 对 5 周交付节奏更友好

### 3.2 Web 框架

- **FastAPI**
- **Pydantic v2**

选择原因：

- 对 JSON API、类型建模、文档生成都很友好
- 适合当前“平台接入 + 管理端 + 学生端 API”结构
- 对后续接口文档和答辩展示也更直观

### 3.3 ORM 与迁移

- **SQLAlchemy 2.0**
- **Alembic**

选择原因：

- 足够稳定，适合 Courseware、CIR、Script、AudioAsset、LearningSession 等关系对象
- 迁移管理清晰，便于团队协作

---

## 4. 异步任务与运行时栈

### 4.1 任务队列

- **Dramatiq + Redis**

选择原因：

- 比 Celery 更轻，学习和接入成本更低
- 足够支撑课件解析、向量索引、TTS 合成、发布装配等后台任务
- 对小团队 Demo 项目更合适

### 4.2 运行边界

同步请求负责：

- 上传接口接入
- 查询课件解析结果
- 获取脚本与播放包
- 问答请求与续讲请求

异步任务负责：

- 课件解析
- CIR 生成后的索引构建
- 批量讲稿生成（如需要）
- 音频合成
- LessonPackage 发布装配

---

## 5. 存储与数据层技术栈

### 5.1 关系型数据库

- **PostgreSQL 16**

这是本项目的主数据库，负责存储：

- Courseware / ParseTask
- CIR / LessonNode / KnowledgeAnchor
- Script / ScriptSection
- AudioAsset / SectionAudio
- PublishedLessonSnapshot / LessonPackage
- QASession / QARecord
- LearningSession / LearningProgress

### 5.2 缓存与中间件

- **Redis**

用途：

- Dramatiq broker
- 热点结果缓存
- 会话型临时状态缓存
- 限流与简单防抖

### 5.3 对象存储

- **开发环境：MinIO**
- **部署环境：S3 兼容对象存储（如云上 S3/OSS/COS）**

用途：

- PPT/PDF 原文件
- 解析中间产物
- 音频文件
- 导出包与演示资源

## 6. 检索与知识点召回技术栈

### 6.1 结论

本项目不建议只用纯向量检索，而应采用 **混合检索**：

- **PostgreSQL Full-Text / BM25 风格词项匹配**
- **pgvector 向量召回**
- **术语字典/别名归一化**
- **重排与 evidence 回链**

### 6.2 推荐组合

- **pgvector**：存储 page/node 级 embedding
- **Postgres Full-Text / trigram**：做术语、关键词、短语精确召回
- **KnowledgeAnchor**：做 page/node/sourceSpan 三元绑定

这样做的原因是：

- 知识点、公式名、课件原句更适合精确匹配
- 同义问法、语义改写适合向量补召回
- 最终需要回链到原页证据，不能只返回语义近似段落

因此本项目检索层定为：

> **Postgres 内统一完成结构数据、全文检索、向量检索与重排前候选筛选**

首版不建议再引入 Elasticsearch / OpenSearch / 独立向量数据库。

---

## 7. LLM 与 TTS 技术栈

### 7.1 LLM 接入

- **托管式 OpenAI-compatible API**
- 后端内部封装一层 **LLM Provider Adapter**

建议做法：

- 首版只接入 **1 家主模型供应商**
- 不做复杂多模型自动路由
- 由成员 C 在 PromptTemplateManager / ModelRouter 上做最薄封装

这样做的好处：

- 兼容现有 LLM 中心架构
- 保留后续替换空间
- 不会在 5 周内把工程复杂度拉高

### 7.2 TTS 接入

- **托管式 TTS 服务 API**
- 后端内部封装一层 **TTS Provider Adapter**

要求：

- 中文语音质量稳定
- 支持分段音频生成
- 支持音色、语速等基本参数控制
- 生成结果可落到 AudioAsset / SectionAudio

这里的原则和 LLM 一样：

- 首版只接 **1 家主 TTS 服务**
- 不做复杂多供应商调度
- 数字人前端层暂不纳入技术栈

## 8. 前端技术栈

### 8.1 推荐栈

- **React 18**
- **TypeScript**
- **Vite**
- **React Router**
- **TanStack Query**
- **Tailwind CSS**

### 8.2 选择原因

- 比 Next.js 更轻，适合 Demo 优先项目
- 适合教师端管理页和学生端学习页两种页面形态
- 便于快速搭建：
  - 上传/解析进度页
  - 脚本编辑页
  - 学生播放页
  - 提问与续讲页

本项目当前不需要 SSR，不建议引入过重的前端工程体系。

---

## 9. 部署与环境技术栈

### 9.1 开发与演示部署

- **Docker Compose**

建议最小运行组合：

- `app-web`
- `app-worker`
- `postgres`
- `redis`
- `minio`

### 9.2 生产/比赛演示策略

如果有云环境，建议：

- Web/Worker 容器化部署
- PostgreSQL / Redis / 对象存储尽量采用托管服务

这样能减少你们在比赛周期里花在基础设施上的时间。

---

## 10. 评测与可观测性技术栈

### 10.1 可观测性

- **Sentry**：异常追踪
- **结构化 JSON 日志**：链路排障
- **Prometheus `/metrics`**：基础指标暴露

建议至少观测：

- 课件解析耗时
- LLM 调用耗时与失败率
- TTS 合成耗时与失败率
- QA 响应时延
- 播放装配成功率

### 10.2 评测工具

- **pytest**
- **JSONL 样本集**
- **离线回归脚本（Python）**

评测重点：

- 知识点召回是否正确
- QA 是否引用正确页码/节点
- 讲稿生成是否结构稳定
- 续讲结果是否回到合理节点

首版不建议引入重型评测平台，先做“够用的回归脚本 + 黄金样本集”。

## 12. 明确不建议的技术路线

以下路线不适合当前阶段：

- 微服务拆分
- Kubernetes
- Kafka / Airflow 等重型中间件
- Elasticsearch + 独立向量库双栈搜索
- 深度依赖 LangChain / LlamaIndex 的复杂编排
- 自建模型推理服务
- 自建 TTS 服务
- Next.js SSR
- 数字人渲染技术栈

原因很简单：这些方案会明显增加复杂度，但对 4 人、5 周、Demo 优先的目标帮助不大。

---

## 13. 最终结论

如果只保留一句话，本项目的推荐技术栈就是：

> **Python + FastAPI + Dramatiq + PostgreSQL/pgvector + Redis + S3对象存储 + 托管式 LLM/TTS + React/Vite**

这套栈的优点不是“最先进”，而是：

- 和现有 LLM 中心架构完全对齐
- 能支撑 CIR、QA、TTS、播放、续讲全链路
- 对小团队和短周期最友好
- 既能做出 Demo，也保留后续继续扩展的空间
