## 5. 核心数据模型设计

### 5.1 对象关系总览
```
Courseware
  -> ParseTask
  -> PageAsset / PageBlock
  -> CIR
      -> Chapter
      -> LessonNode
          -> KnowledgeAnchor
          -> ScriptSection
              -> SectionAudio
  -> ScriptVersion
  -> PublishedLessonSnapshot / LessonPackage
  -> LearningSession
      -> QASession / QARecord
      -> ResumePlan
      -> LearningProgress
```
### 5.2 CIR 定义

CIR（Course Intermediate Representation）是本系统最重要的中间表示。它既是“课程结构 JSON”的正式实现，也是后续讲稿、问答与续讲的共同上下文。

#### CIR 必须满足的规则

- 原文证据层不可丢失：原页文本、图表说明、公式说明、标题等必须可回查；
- 结构标注层可扩展：章节、节点、重点、摘要、依赖关系可以增补；
- 机器补充层可审计：机器生成的摘要、候选讲稿、常见问题必须带来源与版本；
- 所有下游结果必须可回链：脚本、问答、补讲都要能回到页码与证据片段；
- 教师可审核：结构与脚本必须能在教师端查看与修订。

#### CIR 示例

```json
{
  "coursewareId": "cw_001",
  "title": "材料力学-梁弯曲理论",
  "chapters": [
    {
      "chapterId": "chap_01",
      "chapterName": "梁弯曲理论基础",
      "nodes": [
        {
          "nodeId": "node_01",
          "nodeName": "平面假设的定义",
          "pageRefs": [3, 4, 5],
          "keyPoints": [
            "平面假设是梁弯曲理论的基本前提",
            "变形前后的截面保持平面",
            "该假设简化了正应力推导"
          ],
          "summary": "本节点用于建立后续公式推导的前置概念。",
          "anchors": [
            {
              "anchorId": "ka_1001",
              "pageRef": 3,
              "sourceSpan": "平面截面变形后仍保持平面",
              "label": "平面假设"
            }
          ],
          "prerequisiteNodeIds": [],
          "nextNodeId": "node_02"
        }
      ]
    }
  ]
}
```

### 5.3 KnowledgeAnchor 设计

KnowledgeAnchor 是支撑“证据问答”的核心对象，正式定义如下：

```
KnowledgeAnchor
= label
+ pageRef
+ nodeRef
+ sourceSpan
+ aliases(optional)
+ confidence(optional)
```

它的作用是：

保证问答时能返回具体页码；

保证续讲时能回到正确节点；

保证教师审核时知道哪部分是课件原意；

保证知识点召回不只依赖抽象 summary。

### 5.4 发布态与运行态对象

| 维度 | 发布态 | 运行态 |
|---|---|---|
| 代表对象 | `PublishedLessonSnapshot`、`LessonPackage` | `LearningSession`、`QASession`、`ResumePlan` |
| 用途 | 给学生端提供稳定内容快照 | 记录学生当前正在学到哪里、问了什么、接下来怎么讲 |
| 变化频率 | 低，教师发布后稳定 | 高，学习过程中实时变化 |
| 是否可编辑 | 教师重新发布时变更 | 不对学生直接暴露底层对象 |
| 是否参与评测 | 参与讲稿、音频、结构评测 | 参与问答时延、续接效果、学习行为评测 |

### 5.5 核心对象生命周期与状态

为了避免后续接口设计、任务编排和运行态实现各自理解不同，架构层需要把关键对象的生命周期明确为“可查询、可回退、可区分失败类型”的正式约束。

| 对象 | 关键状态 | 状态含义 | 是否可恢复/重试 |
|---|---|---|---|
| `ParseTask` | `created -> processing -> completed / failed / partial_completed` | 课件解析任务从创建到完成；`partial_completed` 表示部分页已形成可用结构，但存在局部识别失败 | `failed`、`partial_completed` 可人工重试或局部补跑 |
| `ScriptVersion` | `draft -> teacher_editing -> confirmed -> superseded` | 脚本先由系统生成草稿，进入教师编辑，确认后成为可发布输入；旧版本被新版本替代时进入 `superseded` | `draft`、`teacher_editing` 可继续修改；`confirmed` 只能通过新版本替代 |
| `PublishedLessonSnapshot` / `LessonPackage` | `assembling -> published / publish_failed / archived` | 发布装配中汇聚 CIR、脚本、音频等资源；成功后形成稳定可学习快照；旧快照归档 | `publish_failed` 可在修复缺失资源后重新装配；`published` 不原地改写 |
| `LearningSession` | `active -> paused -> completed / abandoned` | 学生学习会话的运行状态，支持中断后恢复 | `paused` 可恢复；`abandoned` 不再继续推进 |
| `QASession` / `QARecord` | `opened -> answered -> closed / degraded` | 问答会话和单次问答记录；`degraded` 表示因证据不足、ASR 低置信或弱网而采用降级路径 | `degraded` 可保留记录并允许学生重试或改用文字问答 |
| `ResumePlan` | `generated -> applied / expired / rejected` | 问答后生成续讲方案；被执行后生效，过期或被新方案覆盖后失效 | `expired`、`rejected` 可重新生成 |

### 5.6 生命周期设计约束

- 所有异步对象必须存在“处理中、成功、失败”三类可查询状态，不能只有黑盒任务 ID。
- 允许局部可用的对象，需要显式区分 `partial_completed` 或同等语义状态，避免把“部分成功”误判为彻底失败。
- 发布态对象一旦进入 `published`，即视为稳定快照；后续变更只能通过新版本重新发布，而不是原地覆盖。
- 运行态对象允许高频变化，但必须保证可追踪最近一次有效状态，便于续讲、排障和评测回放。
- 会影响教学连续性的失败态，必须能映射到接口可见状态，而不是只留在内部日志里。

### 5.7 逻辑对象到首批物理表的映射建议

当前仓库已经完成最小 demo 链：`parse -> generateScript -> generateAudio -> publish -> play`。为了让后续 MySQL 落地能直接指导开发，需要把逻辑对象进一步收敛到一组首批物理表，而不是只停留在概念对象层。

推荐优先采用以下映射：

| 逻辑对象 | 首批物理表 | 用途 |
|---|---|---|
| `Courseware` / 教师章节课件 | `course_chapters`、`chapter_ppt_assets` | 保存教师端课程章节树与按章节上传的原始 PPT |
| `ParseTask` | `chapter_parse_tasks` | 保存 `parseId`、任务状态、失败原因、发起人、完成时间 |
| `StructurePreview` / `CIR` / 节点树 | `chapter_parse_results`、`chapter_knowledge_nodes` | 保存解析摘要、页码映射、节点树、知识点来源 |
| `ScriptVersion` | `chapter_scripts`、`chapter_script_sections` | 保存脚本主记录、分段文本、关联知识节点与页范围 |
| `AudioAsset` / `SectionAudio` | `chapter_audio_assets` | 保存音频元数据、来源脚本、状态与对象存储引用 |
| `PublishedLessonSnapshot` / `LessonPackage` | `lessons`、`lesson_units`、`lesson_sections`、`lesson_section_pages`、`lesson_section_anchors`、`lesson_section_knowledge_points` | 保存学生端稳定发布快照，以及章节、页、锚点、知识点的冻结版本 |
| `QASession` / `QARecord` | `qa_sessions`、`qa_messages`、`qa_answers`、`qa_message_knowledge_refs` | 保存多轮问答、结构化回答、知识点回链与推荐续学信息 |
| `LearningProgress` / `ResumePlan` | `student_lesson_progress`、`student_section_progress`、`progress_track_logs`、`resume_records` | 保存总进度、章节进度、续讲建议与历史留痕 |

这组表不是要求一次性全部落完，而是作为后续 repository / models 拆分时的稳定目标面。对当前代码最关键的是：`parse`、`script`、`lesson`、`qa` 的持久化边界已经能直接对上真实表，而不必继续停留在“对象名字存在，但开发时不知道应该落几张表”的状态。

### 5.8 教师生产态与学生发布态必须分表

为了保证“教师反复修改”与“学生稳定学习”不互相污染，架构层需要明确采用两套相关但分离的物理对象：

- 教师生产态：`course_chapters`、`chapter_ppt_assets`、`chapter_parse_tasks`、`chapter_parse_results`、`chapter_knowledge_nodes`、`chapter_scripts`、`chapter_audio_assets`
- 学生发布态：`lessons`、`lesson_units`、`lesson_sections`、`lesson_section_pages`、`lesson_section_anchors`、`lesson_section_knowledge_points`

教师态到发布态的关键回链字段，建议固定为：

- `lesson_units.source_chapter_id`：学生端单元回链教师章节
- `lesson_sections.source_chapter_id`：学生端章节回链教师章节
- `lesson_sections.parse_result_id`：发布快照回链解析结果
- `lesson_sections.ppt_asset_id`：发布快照回链原始 PPT
- `lesson_sections.script_id`：发布快照回链脚本版本
- `lesson_sections.audio_asset_id`：发布快照回链音频版本
- `lesson_section_pages.source_ppt_asset_id`：学生端页内容回链原始页资源来源
- `lesson_section_knowledge_points.source_node_id`：学生端知识点回链教师端知识树节点

这样做的结果是：

- 教师端可以重新上传 PPT、补跑解析、替换脚本；
- 学生端仍只读取某次 `published` 后冻结的快照；
- QA、续讲、进度都可以明确知道自己依赖的是哪一版发布内容，而不是“当前最新生产态”。

### 5.9 首批必须固化的约束与索引

为了让后续接口查询、状态查询与版本冻结不出现歧义，以下约束建议作为首批建模时的强约束：

- `course_chapters(course_id, chapter_code)` 唯一，避免同一课程内章节编码冲突
- `chapter_ppt_assets(chapter_id, version_no)` 唯一，保证同一章节的课件版本线清晰
- `chapter_parse_tasks(parse_no)` 唯一，保证 `parseId` 可稳定回查
- `chapter_parse_results(parse_task_id)` 唯一，保证一条解析任务只对应一份正式结果快照
- `chapter_scripts(script_no)` 唯一，保证 `scriptId` 可稳定回查
- `lessons(course_id, publish_version)` 唯一，保证课程发布版本不冲突
- `lesson_sections(lesson_id, section_code)` 唯一，保证学生端章节编码稳定
- `student_lesson_progress(student_id, lesson_id)` 唯一，保证每个学生在一个 lesson 下只有一条总进度
- `student_section_progress(student_id, lesson_id, section_id)` 唯一，保证章节进度记录不重复
- `qa_sessions(session_no)` 唯一，保证 `sessionId` 可稳定回查

首批必须预留的查询索引，建议包括：

- `chapter_parse_tasks(chapter_id, task_status)`：支撑教师侧查看解析状态
- `chapter_scripts(chapter_id, script_status)`：支撑脚本版本与编辑态查询
- `lessons(course_id, publish_status)`：支撑课程发布记录查询
- `lesson_sections(unit_id, sort_no)`：支撑学生侧按顺序取章节
- `qa_sessions(student_id, lesson_id, updated_at)`：支撑历史问答列表
- `qa_messages(session_id, created_at)`：支撑多轮问答回放

这里的重点不是把完整 DDL 提前塞进架构文档，而是把“哪些关系和约束一旦做错，后续就会反复返工”的部分先固定下来，作为后续 `models.py`、`repository.py` 与 Alembic 迁移的正式参考。
