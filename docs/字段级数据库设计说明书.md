# 字段级数据库设计说明书

## 1. 文档定位

本文档用于定义“基于泛雅平台的 AI 互动智课生成与实时问答系统”的业务数据库设计，覆盖教师端与学生端共用主数据、教师智课生产数据、学生学习进度与 AI 问答数据。

设计依据：

- [超星AI互动智课服务系统开放API设计规范与示例.pdf](C:/Users/HP/Downloads/超星AI互动智课服务系统开放API设计规范与示例.pdf)
- [03-第十七届中国大学生服务外包创新创业大赛A类赛题手册.pdf](C:/Users/HP/Downloads/03-第十七届中国大学生服务外包创新创业大赛A类赛题手册.pdf)
- 当前仓库中的教师端 API、学生端 API、mock 数据和页面交互结构

补充业务约束：

- 教师端后续按“章节”上传 PPT，而不是只按整门课上传
- 每个章节的 PPT 原文件与 LLM 解析内容都要同步发布到学生端对应章节
- 学生端章节必须能直接关联到教师端上传的 PPT、解析结果、讲稿、音频与锚点

## 2. 设计目标

数据库采用一套共享业务库，教师端和学生端按业务域拆分，不做“两套完全独立数据库”。

核心支持以下链路：

1. 外部平台同步学校、用户、课程、班级、选课关系
2. 教师在课程的某一章节上传 PPT
3. 系统对该章节 PPT 进行解析，得到章节知识树、重点、页码、摘要、知识点
4. 系统基于解析结果生成章节讲稿、章节音频、章节锚点
5. 章节内容发布到学生端对应章节
6. 学生进入课程详情页后，按章节学习、问答、续学、记录进度
7. 系统基于问答与进度动态调整学习节奏并记录历史

推荐数据库：

- `MySQL 8`
- 字符集：`utf8mb4`
- 存储引擎：`InnoDB`

## 6. 关键链路与落库方式

### 6.1 平台同步链路

- `/api/v1/platform/syncUser`
  - 写入：`users`、`user_platform_bindings`
- `/api/v1/platform/syncCourse`
  - 写入：`courses`、`course_members`、必要时写 `course_classes`

### 6.2 教师按章节上传 PPT 链路

1. 教师在课程章节上上传 PPT
2. 写入 `chapter_ppt_assets`
3. 发起解析任务，写入 `chapter_parse_tasks`
4. LLM 解析完成后写入：
   - `chapter_parse_results`
   - `chapter_knowledge_nodes`
5. 基于解析结果生成讲稿，写入：
   - `chapter_scripts`
   - `chapter_script_sections`
6. 基于讲稿生成音频，写入：
   - `chapter_audio_assets`
7. 发布到学生端章节，写入：
   - `lesson_sections`
   - `lesson_section_pages`
   - `lesson_section_anchors`

### 6.3 学生学习与问答链路

1. 学生进入课程详情页，读取 `lessons -> lesson_sections`
2. 学习时更新 `student_section_progress`
3. 问答时写入：
   - `qa_sessions`
   - `qa_messages`
   - `qa_message_knowledge_refs`
4. 节奏调整与续学时更新：
   - `student_section_progress`
   - `lesson_section_anchors`

## 7. 关键设计结论

### 7.1 为什么要同时保留 `course_chapters` 和 `lesson_sections`

因为教师端章节是生产视角，学生端章节是发布视角：

- 教师端允许多次修改 PPT、重复解析、生成多个讲稿版本
- 学生端只能看到“某次正式发布”的稳定章节内容

因此不能把教师章节表直接拿来给学生端学习页使用。

### 7.2 为什么 PPT 与解析结果都要单独存

因为你的业务要求是：

- 教师上传的是“原始 PPT”
- 学生学习的是“PPT + LLM 解析内容 + 音频 + 讲稿”

所以必须分离：

- `chapter_ppt_assets` 保存原始文件
- `chapter_parse_results` 保存解析结果
- `lesson_section_pages.parsed_content` 保存发布到学生端的页级解析内容

这样后续即使重新解析，也不会直接污染学生端已发布版本。

### 7.3 为什么问答要拆成会话和消息

因为你当前学生端已经做了：

- 多会话历史问答
- 点击历史会话继续聊天
- 新对话归档旧会话

所以数据库必须按：

- `qa_sessions`
- `qa_messages`

两层建模，而不是只存一张“问答记录表”。
